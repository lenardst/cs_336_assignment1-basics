\section{Problem (unicode1): Understanding Unicode}
\subsection*{(a)}
The expression \texttt{chr(0)} returns the Unicode null character, \texttt{U+0000} (NUL).

\subsection*{(b)}
Its string representation shows an escape sequence like \texttt{'\textbackslash x00'}, while printing it produces a non-printing control character (typically visually empty output).

\subsection*{(c)}
When this character appears in text it is embedded as an invisible null byte that still occupies a position in the string, and some tools/languages may treat it as a terminator and behave unexpectedly.

\section{Problem (unicode2): Unicode Encodings}
\subsection*{(a)}
UTF-8 is usually preferred because it is much more space-efficient for ASCII-heavy text (common on the web), is byte-oriented and backward-compatible with ASCII, and avoids UTF-16 endianness/BOM complications while still representing all Unicode characters.

\subsection*{(b)}
An example is \texttt{b'\textbackslash xc3\textbackslash xa9'} (which is ``é'' in UTF-8): the function is incorrect because it decodes each byte independently, but UTF-8 is variable-length and many characters require combining multiple bytes before decoding.

\subsection*{(c)}
An example invalid two-byte sequence is \texttt{b'\textbackslash xc0\textbackslash xaf'}, which is an overlong UTF-8 encoding and therefore does not decode to any valid Unicode character sequence.

\section{Problem (train\_bpe\_tinystories): BPE Training on TinyStories}

This longest token is reasonable because frequent whitespace-prefixed word fragments are exactly the kind of units BPE tends to merge.

\section{Problem (tokenizer\_experiments): Experiments with Tokenizers}
\subsection*{(a)}
On 10 sampled TinyStories documents, the TinyStories tokenizer achieved 4.150 bytes/token (7412 bytes over 1786 tokens). On 10 sampled OpenWebText documents, the OpenWebText tokenizer achieved 4.680 bytes/token (31487 bytes over 6728 tokens), indicating better compression on its in-domain sample.

\subsection*{(b)}
Tokenizing the OpenWebText sample with the TinyStories tokenizer produced 3.160 bytes/token (31487 bytes over 9963 tokens), versus 4.680 bytes/token with the OpenWebText tokenizer on the same sample. This means the TinyStories tokenizer produced many more tokens (worse compression), with a cross-minus-native difference of $-1.520$ bytes/token.

\subsection*{(c)}
Measured throughput was about 4.12 MB/s for the TinyStories tokenizer and 1.92 MB/s for the OpenWebText tokenizer (benchmarked on about 5.0M bytes each). At those rates, tokenizing the 825GB Pile is estimated to take about 55.67 hours (2.32 days) with the TinyStories tokenizer or 119.08 hours (4.96 days) with the OpenWebText tokenizer.

\subsection*{(d)}
\texttt{uint16} is appropriate because both vocabularies are below 65,536 entries (10K and 32K), so every token ID fits exactly in 16 bits. Using \texttt{uint16} therefore halves storage relative to \texttt{int32} while preserving all IDs.
