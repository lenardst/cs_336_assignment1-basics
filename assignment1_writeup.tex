\section{Problem (unicode2): Unicode Encodings}
\subsection*{(a)}
UTF-8 is usually preferred because it is much more space-efficient for ASCII-heavy text (common on the web), is byte-oriented and backward-compatible with ASCII, and avoids UTF-16 endianness/BOM complications while still representing all Unicode characters.

\subsection*{(b)}
An example is \texttt{b'\textbackslash xc3\textbackslash xa9'} (which is ``é'' in UTF-8): the function is incorrect because it decodes each byte independently, but UTF-8 is variable-length and many characters require combining multiple bytes before decoding.

\subsection*{(c)}
An example invalid two-byte sequence is \texttt{b'\textbackslash xc0\textbackslash xaf'}, which is an overlong UTF-8 encoding and therefore does not decode to any valid Unicode character sequence.

\subsection*{(a)}
On 10 sampled TinyStories documents, the TinyStories tokenizer achieved 4.150 bytes/token (7412 bytes over 1786 tokens). On 10 sampled OpenWebText documents, the OpenWebText tokenizer achieved 4.680 bytes/token (31487 bytes over 6728 tokens), indicating better compression on its in-domain sample.

\subsection*{(b)}

\subsection*{(c)}

\subsection*{(d)}
\texttt{uint16} is appropriate because both vocabularies are below 65,536 entries (10K and 32K), so every token ID fits exactly in 16 bits. Using \texttt{uint16} therefore halves storage relative to \texttt{int32} while preserving all IDs.

\subsection*{(a)}
Trainable parameters (no linear biases; \texttt{RMSNorm} has only a scale vector):
\[
\text{total}
= (50{,}257\cdot1{,}600)
+ 48\left(4\cdot1{,}600^2 + 3\cdot1{,}600\cdot4{,}288 + 2\cdot1{,}600\right)
+ 1{,}600
+ (1{,}600\cdot50{,}257)
\]
\[
= 80{,}411{,}200 + 1{,}479{,}628{,}800 + 1{,}600 + 80{,}411{,}200
= 1{,}640{,}452{,}800
\]

If each is stored as a single-precision floating point, the model needs $32 \times 1{,}640{,}452{,}800 = 52{,}494{,}489{,}600\text{ bits} \approx 6.56\text{ GB}$. 6.56~GB are required just to load the model.

\subsection*{(b)}
For one \texttt{TransformerBlock}, let \(B=\) batch size, \(T=\) sequence length, \(d=1600\), \(h=25\), \(d_k=d/h=64\), \(d_{\text{ff}}=4288\). The 9 matrix multiplications are:
\begin{itemize}
    \item Q projection: \([B,T,d]\times[d,d]\to[B,T,d]\)
    \item K projection: \([B,T,d]\times[d,d]\to[B,T,d]\)
    \item V projection: \([B,T,d]\times[d,d]\to[B,T,d]\)
    \item Attention scores \(QK^\top\): \([B,h,T,d_k]\times[B,h,d_k,T]\to[B,h,T,T]\)
    \item Attention-value product: \([B,h,T,T]\times[B,h,T,d_k]\to[B,h,T,d_k]\)
    \item Output projection \(W_O\): \([B,T,d]\times[d,d]\to[B,T,d]\)
    \item FFN branch \(W_1\): \([B,T,d]\times[d,d_{\text{ff}}]\to[B,T,d_{\text{ff}}]\)
    \item FFN branch \(W_3\): \([B,T,d]\times[d,d_{\text{ff}}]\to[B,T,d_{\text{ff}}]\)
    \item FFN output \(W_2\): \([B,T,d_{\text{ff}}]\times[d_{\text{ff}},d]\to[B,T,d]\)
\end{itemize}

For \texttt{TransformerLM} outside the \texttt{TransformerBlock}s:
\begin{itemize}
    \item Final output projection: \([B,T,d]\times[d,\text{vocab\_size}]\to[B,T,\text{vocab\_size}]\)
\end{itemize}

One forward pass needs \(9\times48+1=433\) matrix multiplications.

General rule for matrix multiplication FLOPs:
\[
([m,n]\times[n,p]) \Rightarrow 2mnp \text{ FLOPs}
\]

Using \(B=1\), \(T=1024\), \(d=1600\), \(h=25\), \(d_k=64\), \(d_{\text{ff}}=4288\), \(\text{vocab\_size}=50{,}257\):
\begin{itemize}
    \item Q, K, V, \(W_O\): each \(2BTdd=5{,}242{,}880{,}000\)
    \item \(QK^\top\), attention-value: each \(2BhTTd_k=3{,}355{,}443{,}200\)
    \item \(W_1, W_3, W_2\): each \(2BTdd_{\text{ff}}=14{,}050{,}918{,}400\)
\end{itemize}

Per-block FLOPs:
\[
4(5{,}242{,}880{,}000) + 2(3{,}355{,}443{,}200) + 3(14{,}050{,}918{,}400)
= 69{,}835{,}161{,}600
\]

All 48 blocks:
\[
48\times69{,}835{,}161{,}600
= 3{,}352{,}087{,}756{,}800
\]

Final output projection FLOPs:
\[
2BTd\cdot\text{vocab\_size}
= 2\cdot1\cdot1024\cdot1600\cdot50{,}257
= 164{,}682{,}137{,}600
\]

Total FLOPs in one forward pass:
\[
3{,}352{,}087{,}756{,}800 + 164{,}682{,}137{,}600
= \boxed{3{,}516{,}769{,}894{,}400}
\]

\subsection*{(c)}
The PositionwiseFeedForward requires a lot of FLOPs (14,050,918,400 x 3 = 42,152,755,200) which is more than 4 times more than the CausalMultiHeadSelfAttention (3,355,443,200 * 2 + 5,242,880,000 = 11,953,766,400).
